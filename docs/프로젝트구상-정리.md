# 금융 분석 AI 시스템 - 프로젝트 구상 정리

## 1. 프로젝트 개요

**프로젝트명**: 금융 분석 AI 시스템 (Spring Boot + RAG + LLM)

**목표**: 사용자가 금융 질문(종목, 실적, 이벤트 등)을 하면 **문서 / 가격 / 재무 데이터를 검색 -> 분석 -> 근거 포함 답변**을 생성하는 시스템. "주가 예측 AI"가 아니라 **근거 기반 해석 AI**.

**포트폴리오 + 앱 출시** 가능성을 모두 염두에 둔 설계.

### 예시 질문

- "엔비디아 실적 이후 왜 삼성전자보다 하이닉스가 더 강하게 반응했지?"
- "유가 급등이 정유, 항공, 조선, 방산에 미치는 영향 경로를 설명해줘"
- "이 종목 최근 2주 투자 포인트를 공시, 뉴스, 실적 기준으로 정리해줘"
- "이 기업 리스크를 긍정/부정 시나리오로 나눠줘"

## 6. RAG 파이프라인 상세

```
질문 입력
   |
   v
[1] Query Processing --- 의도 분류 + Metadata Filter 생성 (ticker, market, sourceType, 날짜)
   |
   v
[2] SourceType Fallback Retrieval
   |  질문 유형별 우선순위:
   |  EARNINGS_INTERPRETATION -> REPORT > FILING > NEWS
   |  EVENT_IMPACT_ANALYSIS   -> NEWS > REPORT > FILING
   |  RISK_SUMMARY            -> FILING > REPORT > NEWS
   |  STOCK_ANALYSIS          -> NEWS > REPORT > FILING
   |  최소 결과 수 미달 시 다음 sourceType으로 확장
   |
   v
[3] Keyword Retrieval (PostgreSQL FTS, tsvector/ts_rank_cd)
   +
   Vector Retrieval (pgvector cosine similarity)
   |
   v
[4] Hybrid Ranking --- final_score = alpha * vector_score + beta * keyword_score
   |
   v
[5] Neighbor Chunk Expansion --- 검색된 chunk의 앞/뒤 chunk 문맥 확장 (window=1)
   |
   v
[6] Adjacent Hit Merge --- 같은 문서 내 인접 hit를 하나의 context로 병합, 중복 제거
   |
   v
[7] Lightweight Reranker --- merged context 단위로 재정렬
   |
   v
[8] Final Context -> LLM 전달
   |
   v
[9] Retrieval Trace 수집 --- 전 단계 결과를 JSON으로 기록 -> DB 저장 -> UI 시각화
```

## 7. DB 테이블 설계 (Flyway V1~V10)

### V1: 핵심 테이블

| 테이블 | 역할 | 주요 컬럼 |
|---|---|---|
| `users` | 사용자 | id(UUID), email, password_hash, display_name, role, active |
| `documents` | 문서 원문 | id, source_type, title, url, published_at, cleaned_text, summary, ticker, market, metadata_json |
| `document_chunks` | 문서 chunk + embedding | id, document_id(FK), chunk_index, chunk_text, token_count, embedding vector(1536) |

### V2: 분석 테이블

| 테이블 | 역할 | 주요 컬럼 |
|---|---|---|
| `analysis_sessions` | 분석 세션 | id, user_id(FK), title, query_text, query_intent, analysis_mode |
| `analysis_results` | 분석 결과 | id, session_id(FK), summary, full_text, confidence_score, uncertainty_note, citations_json, tool_trace_json, trace_json |

### V3: 시장 데이터

| 테이블 | 역할 | 주요 컬럼 |
|---|---|---|
| `price_daily` | 일별 가격 | id, ticker, market, trade_date, open/high/low/close_price, volume, change_rate |
| `financial_metrics` | 분기별 재무지표 | id, ticker, market, fiscal_year/quarter, period_type, revenue, operating_profit, net_income, eps, roe, debt_ratio |

### 주요 인덱스

- `document_chunks`: IVFFlat vector index (cosine, lists=100)
- `document_chunks`: GIN index (tsvector)
- `documents`: ticker + market, source_type, published_at
- `price_daily`: ticker + market + trade_date (unique)
- `financial_metrics`: ticker + market + fiscal_year + quarter + period_type (unique)

## 10. PIPELINE vs TOOL CALLING 비교

| 항목 | PIPELINE | TOOL CALLING |
|---|---|---|
| 흐름 | 고정 (결정적) | 동적 (LLM이 선택) |
| 검색 | Hybrid RAG 파이프라인 | LLM이 필요한 도구 호출 |
| 재현성 | 높음 | 낮음 (LLM 판단에 의존) |
| Traceability | 전 단계 trace 기록 | tool 호출 기록 |
| 유연성 | 낮음 (고정 흐름) | 높음 (질문별 적응) |
| 적합 상황 | 표준 분석, 안정적 결과 | 복합 질문, 동적 데이터 조합 |

### Tool Calling 도구 3종

| 도구 | 역할 |
|---|---|
| `searchRelevantContexts(query, ticker, market, from, to)` | 문서 검색 (RAG) |
| `getPriceSummary(ticker, market, from, to)` | 가격 데이터 요약 |
| `getFinancialSummary(ticker, market)` | 재무 지표 요약 |

## 11. Retrieval Observability (관측성)

각 분석 세션에 대해 retrieval 전 과정을 trace로 기록:

| 단계 | 기록 내용 |
|---|---|
| Source Fallback | 어떤 sourceType 순서로 검색했는지, 각 몇 건 |
| Hybrid Results | chunk별 vector score, keyword score, final score |
| Expanded Results | 중심 chunk + 포함된 chunk index 목록 |
| Merged Results | 병합된 문맥 단위, 포함 chunk, 대표 점수 |
| Reranked Results | 최종 rerank 점수, expanded text |

저장: `analysis_results.trace_json` (JSON)
시각화: 프론트 콘솔의 Retrieval Trace 영역

## 12. Spring -> Python 마이크로서비스 경계선

### Spring만으로 충분한 영역

- API 서버 (인증, 세션, CRUD)
- LLM orchestration (프롬프트, tool calling)
- RAG 파이프라인 (retrieval, merge, rerank)
- 간단한 수치 계산 (수익률, 변동률, 단순 ratio)

### Python을 붙여야 하는 경계선

- 대량 시계열 지표 계산 (RSI, MACD, Bollinger, rolling stats)
- 종목 수백 개 일괄 factor 계산
- 이벤트 스터디 (이벤트 전후 수익률 분석)
- 백테스트
- 상관계수/회귀/클러스터링

### 목표 구조

```
Spring Boot = 메인 앱 / 오케스트레이터
Python (FastAPI) = 수치 분석 엔진 (gRPC or REST 연결)
```
